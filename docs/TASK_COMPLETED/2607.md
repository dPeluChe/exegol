## 2026-07-04: T88v2 — Evaluator Gate (two-pass judge + score distribution + ship/hold/retry)

Branch: `feat/wtc-t88v2-evaluator-gate`. Source: Paseo "Ralph loop" pattern (base T88, never
shipped) + 2026-07 agent-eval/theloop review (v2 upgrade). Since base T88 didn't exist yet,
this PR ships the full evaluator step type plus the v2 upgrades in one pass — the backlog
entry only ever described the v2-upgraded shape.

### Types (`packages/shared/src/types/pipeline.ts`)
- `PipelineStepDef.evaluator?: EvaluatorStepDef` — marks a step as a judge gate instead of a
  spawned agent: `acceptanceCriteria`, `onPassNext`/`onFailNext` (step indices), `maxLoops`,
  `judgeCalls` (default 3), `gatePolicy` (`shipThreshold`/`holdThreshold`, defaults 0.7/0.4).
- `EVALUATOR_HARD_MAX_LOOPS = 10` — hard cap regardless of a step's configured `maxLoops`.
- `EvaluatorVerdict` on `PipelineStepResult.verdict`: `decision` (ship/hold/retry), `scores`
  (the full distribution, not just the average), `avgScore`, `feedback`, `costUsd`.
- `{{retryFeedback}}` added to `buildStepPrompt`'s template variables.

### Judge (`main/pipeline/evaluator.ts`, pure — no DB access)
- **Two-pass per call**: pass 1 asks Haiku to adversarially *describe* what the diff actually
  does (no verdict); pass 2 renders a 0-1 score from that description against the acceptance
  criteria. Separating description from verdict is the whole point — it stops the judge from
  rationalizing a lenient score against the raw diff.
- **N independent calls** (default 3, configurable) run concurrently → a real score
  distribution, not a single PASS/RETRY. A failed call scores 0 rather than aborting the gate.
- **Gate policy**: avgScore ≥ shipThreshold → ship; ≥ holdThreshold → hold (pause for a human);
  else → retry. `decide()` is exported and unit-tested in isolation from the network calls.
- **Cost tracking**: reads `usage.input_tokens`/`output_tokens` from the Anthropic response,
  prices at an approximate Haiku rate — flagged below, not a real price table.
- Retry feedback is derived from the lowest-scoring call's pass-1 description (no extra API
  call needed) and fed to the loop-back step as `{{retryFeedback}}`.

### Routing (`main/pipeline/evaluator-step-handler.ts`, `executor.ts`)
- Evaluator steps resolve synchronously inside `advanceStep` (judge calls, no spawned agent,
  no exit-code callback) — routed to `handleEvaluatorStep` instead of the normal agent-spawn
  path, mirroring `handleStepComplete`'s ship/hold/retry-style branching but driven by the
  verdict instead of an exit code.
- ship → advance to `onPassNext` (default next step) or complete the run.
- hold → `pauseRun` with the avg score in the reason (existing running→paused transition).
- retry → loop back to `onFailNext` (`iterationCount` incremented); pauses if no loop target
  is configured or the hard-max/`maxLoops` cap is hit.

- `PipelineTemplateEditor.tsx`: "Evaluator gate" checkbox per step toggles between the normal
  agent fields and a new `EvaluatorStepFields.tsx` (acceptance criteria, on-pass/on-retry step
  pickers, max loops, judge calls) — extracted to its own file to keep the editor under the

- `PipelineRunView.tsx`: evaluator steps show a "evaluator gate" badge instead of

## 2026-07-05: T126 — Memory Salience v2 (Decay + Reinforcement + Supersession)

Branch: `feat/t126-memory-salience-v2`.

### Implementation
- New `memory/salience.ts`: `computeSalience()` implements
  `salience = similarity × log(reinforcement_count + 1) × exp(-0.693 × days_ago / 30)`
  (30-day half-life). `relevanceScore` stands in for `similarity` (salience is computed
  at rest, not per-query). `reinforcement_count` defaults to 1 (the initial observation) so
  a never-reinforced fact still has nonzero salience instead of being zeroed by `log(1)=0`.
  Also hosts the shared `textSimilarity`/`normalizeForDedup` Jaccard helper (moved out of
  `extractor.ts` to avoid duplicating it) and `classifyObservation()`.
- Migration `w2b_001_memory_salience_v2` (`db/migration-sets/wave2-knowledge.ts`): adds
  `reinforcement_count`, `last_reinforced_at`, `superseded_by` to `memories`.
- `memory/store.ts`:
  - `reinforceMemory(id)` bumps `reinforcement_count` + `last_reinforced_at`.
  - `supersedeMemory(oldId, data)` inserts a new row and sets the old row's `superseded_by`
    — the old row is never overwritten or deleted.
  - `observeMemory(data)` classifies a newly-observed fact against active
    (`superseded_by IS NULL`) memories in the same project+category via
    `classifyObservation` (Jaccard similarity ≥0.8 → reinforce, 0.35-0.8 → supersede,
    below → create).
  - `getMemoriesForInjection` now ranks by `computeSalience` (computed in JS — sqlite has
    no LN/EXP without an extension) instead of the old static 3-factor score, and excludes
    superseded facts.
  - `listMemories`/`searchMemories` exclude superseded facts by default
    (`listMemories` takes an `includeSuperseded` opt-out for a future history view).

  fact instead of a flat dedup-then-create. Added an anti-ephemeral guard (`isEphemeral`)
  that drops transient progress-noise lines (`Building...`, `42% complete`, bare timestamps,
  etc.) before they can slip through the looser convention/preference regexes.

## 2026-07-04: T129 — Oplog v2: Git-Tree Snapshots per Agent Turn

Branch: `feat/wtc-pipelines`. Source: GitButler `crates/gitbutler-oplog/` model (see
`docs/RESEARCH/COMPETITIVE_REVIEW_2026_07.md` §4).

### Rust core (`packages/core-rust/src/git/oplog.rs`, `types.rs`)
- Hidden ref chain `refs/exegol/oplog` — outside `refs/heads/*`, never touches HEAD, the
  index file, or the visible branch history; its own independent parent-child commit chain.
- **Unmaterialized pattern**: `prepare_turn_snapshot()` stages the current worktree+index into
  an in-memory `Index` (`add_all`/`update_all`, never `index.write()`) and writes only a tree
  object to the odb — the real `.git/index` file is untouched. `commit_turn_snapshot()` commits
  that tree onto the hidden chain only when called — callers skip the call entirely on turn
  failure, so nothing is ever written for a failed turn.
- **OperationKind trailers**: commit message carries `operation` / `agent-id` / `provider` /
  `turn-index` trailers + free-text description, parsed back out by `list_oplog_snapshots()`.
- **Anti-GC**: `update_hidden_ref()` moves the ref and forges an explicit reflog entry
  (GitButler `reflog.rs` trick) recording the move independent of `core.logAllRefUpdates`.
- `restore_oplog_snapshot()` reuses the existing `revert_to_snapshot()` — a snapshot's tree is
  resolvable by SHA regardless of hidden-ref reachability once committed to the odb. Fixed a
  latent bug in `revert_to_snapshot`'s checkout (`remove_untracked(true)` now set, so restoring
  to an earlier snapshot actually removes files added in later turns).
- 2 new unit tests (`hidden_ref_never_touches_branches`, `chain_links_and_trailers_roundtrip`)
  on top of the existing suite — 45/45 pass. `cargo check && cargo test && cargo clippy` clean.

### Trigger — pipeline steps, stub turn boundary
- Per the stub note (T123 not yet merged), turn boundaries use **process-exit**: a pipeline
  step's agent process finishing is the "turn." `main/pipeline/oplog-snapshots.ts` wraps
  prepare/commit with try/catch — a broken snapshot never breaks pipeline execution.
- `executor.ts`: `prepareStepSnapshot()` called right before spawn, tree sha kept in a new
  `pendingSnapshots: Map<agentId, treeSha>` alongside `activeAgents`/`idleTimers`.
- `pipeline-step-handler.ts`: `handleStepComplete()` commits the pending tree with
  `operation: "PipelineStep"` only when `exitCode === 0`; always deletes the pending entry
  either way (discard-on-failure).
- Scoped to pipelines because `main/agents/manager.ts` (any-agent turn boundary) is WT-A's
  write set — general per-agent-turn wiring (not just pipeline steps) needs to happen once

### IPC + UI
