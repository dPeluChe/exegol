## 2026-07-05: T125 — Hybrid Search RRF (FTS5 + Ollama vector)

Branch: `feat/t125-hybrid-search-rrf`.

### Implementation
- `db/queries/search.ts`: `bm25(search_index, 4.0, 1.0)` weighting (title/body — the schema
  has no `path` column so qmd's third weight is dropped); `search()` gained `entityType` filter.
  New `hybridSearch()` fuses the FTS5 keyword list with an Ollama-embedding similarity pass
  via Reciprocal Rank Fusion (qmd's `hybridQuery`/`blend` formula):
  `score += weight / (60 + rank + 1)`, +0.05 bonus for rank 0, +0.02 for rank 1. Keyword list
  weight 2.0 (literal/"original query" match), vector list weight 1.0. Strong-BM25 probe
  (`|score| >= 8`) skips the vector pass — and its Ollama round-trips — when the keyword hit
  is already decisive. Vector pass batches the top 25 keyword candidates through
  `generateEmbeddingsBatch` (one round-trip, not one call per candidate).
- `SearchEntityType` (`packages/shared/src/types/search.ts`) gained `"memory"`.
- `memory/store.ts`: `createMemory`/`deleteMemory` now index/deindex into `search_index`
  (`memory:<id>` entity ids). `searchMemories` is now async, calls `hybridSearch` scoped to
  `entityType: "memory"`, falls back to the old LIKE scan only when FTS5 yields zero tokens.
- `ipc/procedures/memory.ts` `search` and `ipc/procedures/search.ts` `query` (global search
  section) both accept optional `ollamaUrl`/`ollamaModel` and default to
  `DEFAULT_SETTINGS.ollamaUrl/ollamaModel` — hybrid search runs opportunistically; keyword-only
  when Ollama is unreachable (fails fast, doesn't block the query).

  confirmed present on `main` too — unrelated to this change).

  existing coverage untouched).
