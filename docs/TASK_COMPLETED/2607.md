## 2026-07-04: T130 ‚Äî Pipeline Evidence (Verifiable Artifacts per Step)

Branch: `feat/wtc-t130-pipeline-evidence`. Source: Google Antigravity "Artifacts" gap analysis
(`docs/RESEARCH/COMPETITIVE_REVIEW_2026_07.md`).

### What already existed
- Per-step diff + scrollback tail were already captured in `PipelineStepResult` by
  `pipeline-step-handler.ts` (`diffSummary`/`outputSummary`) ‚Äî no new capture plumbing needed
  there.

### New: `main/pipeline/evidence.ts`
- `summarizeStepDiff()` ‚Äî reuses the exact Haiku pattern from `diff-ai.ts`'s
  `suggestCommitMessage` (raw fetch to `api.anthropic.com`, 20KB diff truncation, best-effort:
  returns `""` on no API key / empty diff / network failure, never throws).
- `attachStepScore()` ‚Äî reads `agent_scores.overall_score` for the step's agent id via the
  existing `getAgentScore` query; `null` if scoring hasn't landed yet (best-effort, no wait).
- `buildRunReport()` ‚Äî per-step markdown (status, provider, score, duration, AI summary,
  collapsible diff) suitable for pasting into a PR description.

### Wiring
- `pipeline-step-handler.ts`: on step completion, `summarizeStepDiff`/`attachStepScore` run
  alongside the existing scrollback/diff capture and land in two new optional
  `PipelineStepResult` fields (`aiSummary`, `score`) ‚Äî `packages/shared/src/types/pipeline.ts`.
  AI summary is skipped on step failure (nothing useful to summarize).
- `ipc/procedures/pipeline.ts`: new `pipeline.exportRunReport` query.
- `use-trpc-pipeline.ts`: new `useExportRunReport()` mutation (lazy ‚Äî only fetched on export).
- `PipelineRunView.tsx`: score badge + AI summary line per step, new "Diff" collapsible
  (alongside the existing "Output" one), header "Export Report" button that copies the
  markdown report to the clipboard.

### Not done (flagged, out of scope for this PR)
- Browser-pane screenshot evidence (scope item) ‚Äî no existing per-step screenshot capture

- Evaluator verdict field ‚Äî depends on T88v2 (this WT-C group's next task); `score`/`aiSummary`
  ship now, verdict will be added to the same `PipelineStepResult` shape once T88v2 lands.

## 2026-07-05: T127 ‚Äî Progressive Disclosure Skills

Branch: `feat/t127-progressive-disclosure-skills`.

### Implementation
- `Skill` type (`packages/shared/src/types/skill.ts`) gained `always: boolean` ‚Äî the
  escape hatch for tiny critical skills that should always be fully inlined.
- `skills/loader.ts` parses `always: true` from frontmatter (`meta.always`).
- `agents/spawn-context.ts`: new `buildSkillContext(projectPath, requestedNames?)` replaces
  the old full-body injection. `always: true` skills are always fully inlined
  (`## Skill: <name>\n\n<content>`) regardless of selection ‚Äî that's the point of the
  escape hatch. Explicitly-selected non-always skills get a metadata-only pointer instead
  (`- **name**: description ‚Äî read \`filePath\` when this task matches.`), ~100 tokens/skill
  instead of the full body. The CLI is expected to use its own read tool to load the file
  when the task matches.
- `buildSpawnContext` now always calls `buildSkillContext` (not gated on
  `config.skillNames?.length` ‚Äî always-on skills need to show up even with no explicit
  selection).
- Review fixes (post-merge of PR #43): `EXEGOL_SKILLS_DIR` test isolation override,
  bin-availability cache (5 min TTL), empty-description pointer fallback, warn on
  unrecognized `always` values.

### Quality gates
- Biome clean; tsc node clean; `spawn-context.test.ts` 3/3 pass (isolated from the
  developer's global skills dir).
