## 2026-07-05: T125 — Hybrid Search RRF (FTS5 + Ollama vector)

Branch: `feat/t125-hybrid-search-rrf`.

### Implementation
- `db/queries/search.ts`: `bm25(search_index, 4.0, 1.0)` weighting (title/body — the schema
  has no `path` column so qmd's third weight is dropped); `search()` gained `entityType` filter.
  New `hybridSearch()` fuses the FTS5 keyword list with an Ollama-embedding similarity pass
  via Reciprocal Rank Fusion (qmd's `hybridQuery`/`blend` formula):
  `score += weight / (60 + rank + 1)`, +0.05 bonus for rank 0, +0.02 for rank 1. Keyword list
  weight 2.0 (literal/"original query" match), vector list weight 1.0. Strong-BM25 probe
  (`|score| >= 8`) skips the vector pass — and its Ollama round-trips — when the keyword hit
  is already decisive. Vector pass batches the top 25 keyword candidates through
  `generateEmbeddingsBatch` (one round-trip, not one call per candidate).
- `SearchEntityType` (`packages/shared/src/types/search.ts`) gained `"memory"`.
- `memory/store.ts`: `createMemory`/`deleteMemory` now index/deindex into `search_index`
  (`memory:<id>` entity ids). `searchMemories` is now async, calls `hybridSearch` scoped to
  `entityType: "memory"`, falls back to the old LIKE scan only when FTS5 yields zero tokens.
- `ipc/procedures/memory.ts` `search` and `ipc/procedures/search.ts` `query` (global search
  section) both accept optional `ollamaUrl`/`ollamaModel` and default to
  `DEFAULT_SETTINGS.ollamaUrl/ollamaModel` — hybrid search runs opportunistically; keyword-only
  when Ollama is unreachable (fails fast, doesn't block the query).

### Quality gates
- Biome clean, tsc node clean, tsc web clean (5 pre-existing `@radix-ui/react-dialog` errors
  confirmed present on `main` too — unrelated to this change).

## 2026-07-05: T126 — Memory Salience v2 (Decay + Reinforcement + Supersession)

Branch: `feat/t126-memory-salience-v2`.

### Implementation
- New `memory/salience.ts`: `computeSalience()` implements
  `salience = similarity × log(reinforcement_count + 1) × exp(-0.693 × days_ago / 30)`
  (30-day half-life). `relevanceScore` stands in for `similarity` (salience is computed
  at rest, not per-query). `reinforcement_count` defaults to 1 (the initial observation) so
  a never-reinforced fact still has nonzero salience instead of being zeroed by `log(1)=0`.
  Also hosts the shared `textSimilarity`/`normalizeForDedup` Jaccard helper (moved out of
  `extractor.ts` to avoid duplicating it) and `classifyObservation()`.
- Migration `w2b_001_memory_salience_v2` (`db/migration-sets/wave2-knowledge.ts`): adds
  `reinforcement_count`, `last_reinforced_at`, `superseded_by` to `memories`.
- `memory/store.ts`:
  - `reinforceMemory(id)` bumps `reinforcement_count` + `last_reinforced_at`.
  - `supersedeMemory(oldId, data)` inserts a new row and sets the old row's `superseded_by`
    — the old row is never overwritten or deleted.
  - `observeMemory(data)` classifies a newly-observed fact against active
    (`superseded_by IS NULL`) memories in the same project+category via
    `classifyObservation` (Jaccard similarity ≥0.8 → reinforce, 0.35-0.8 → supersede,
    below → create).
  - `getMemoriesForInjection` now ranks by `computeSalience` (computed in JS — sqlite has
    no LN/EXP without an extension) instead of the old static 3-factor score, and excludes
    superseded facts.
  - `listMemories`/`searchMemories` exclude superseded facts by default
    (`listMemories` takes an `includeSuperseded` opt-out for a future history view).
- `memory/extractor.ts`: `extractAndStoreMemories` now calls `observeMemory` per extracted
  fact instead of a flat dedup-then-create. Added an anti-ephemeral guard (`isEphemeral`)
  that drops transient progress-noise lines (`Building...`, `42% complete`, bare timestamps,
  etc.) before they can slip through the looser convention/preference regexes.
- Considered adding memU's `tool`/`behavior` categories per the task's "consider adding" note,
  but deferred: `MEMORY_CATEGORIES`'s CHECK constraint can't be widened with a plain
  `ALTER TABLE` in sqlite (needs a table-rebuild migration) — not worth the risk for an
  optional scope item. Left as a follow-up if a real need shows up.

### Quality gates
- Biome clean, tsc node clean, tsc web clean (same 5 pre-existing `@radix-ui/react-dialog`
  errors present on `main`).
- `bun test src/main/memory/` — 28 pass, 0 fail (new `salience.test.ts` covers decay,
  reinforcement, and the reinforce/supersede/create classification).

## 2026-07-05: T140 — Project Knowledge Node

Branch: `feat/t140-knowledge-node` (built on top of T126 — depends on `observeMemory`/
`computeSalience` for the MEMORY.md bridge).

### Implementation — storage model exactly as decided in the task spec
New `main/knowledge/` module:
- `paths.ts` — path helpers for `.exegol/knowledge/{PROJECT,DIGEST,MEMORY}.md`;
  `ensureDigestGitignored` idempotently appends `.exegol/knowledge/DIGEST.md` to `.gitignore`
  (derivable/high-churn, never pollutes PRs).
- `brief.ts` — `PROJECT.md`: user-editable, section-structured (`What it does` /
  `Where it's going` / `Key decisions`) to minimize merge conflicts. `ensureProjectBrief`
  creates the template once and never overwrites; `writeProjectBrief` is the explicit
  full-overwrite used by the UI's Save button (the user's intent is explicit there).
- `digest.ts` — `DIGEST.md`: tries an external `trs digest <path>` binary first (any
  failure — including today's near-universal case where no such subcommand exists — falls
  through silently), else an internal summarizer (top-level directory/file-count breakdown +
  package.json dependency list).
- `staleness.ts` — digest freshness: each generated digest is stamped with an HTML-comment
  marker holding the git HEAD sha it was built from; stale = ≥10 commits behind (or missing
  entirely). No new DB table — the marker lives inline in the gitignored file.
- `managed-block.ts` — delimited `<!-- exegol:knowledge:begin/end -->` block upserted into
  whichever of `AGENTS.md`/`CLAUDE.md` exist at the project root (creates `AGENTS.md` if
  neither does). Exegol only ever touches content between the markers.
- `memory-bridge.ts` — `MEMORY.md`: `syncMemoryBridge` distills the top-salience active DB
  facts (via T126's `computeSalience`) into a committed file; `importMemoryBridgeAsSeed`
  parses it back and routes every fact through `observeMemory` (safe to re-run — never
  duplicates).
- `context.ts` — `buildKnowledgeContext`: progressive-disclosure pointer block (same
  mechanism as T127) listing whichever knowledge files exist, instructing the CLI to read
  them on demand — never inlines full file contents into the prompt.

### Wiring
- `agents/spawn-context.ts`: new `knowledgeContext` field, folded into `contextPrefix`
  alongside memory/MCP/skill context (non-fatal try/catch, same pattern as the others).
- `pipeline/context.ts`: `buildStepPrompt` gained an optional `knowledge` var and
  `{{knowledge}}` template interpolation. `pipeline/executor.ts`'s `advanceStep` resolves the
  run's project and passes `buildKnowledgeContext(project.path)` in.
- New `ipc/procedures/knowledge.ts` (`knowledge.get/saveBrief/refreshDigest/syncMemoryBridge/importMemoryBridgeAsSeed`),
  registered in `ipc/router.ts`.
- New renderer `sections/KnowledgeSection.tsx` + `hooks/use-trpc-knowledge.ts`; new "Knowledge"
  sub-tab wired into `WorkspaceTabs.tsx` (`ProjectSubTab`/`WorkspaceSection` + `PROJECT_SUBS`)
  and `WorkspaceView.tsx` (lazy import + conditional render), following the same pattern as
  `MemorySection`.
